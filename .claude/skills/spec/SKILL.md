# 요구사항 설계

반복 질문으로 요구사항을 구체화한 뒤, 문서를 생성합니다.

## 1단계: 반복 질문

### 사전 탐색 (기능 확장 시)

질문을 시작하기 전에, 기존 기능을 확장하는 경우 다음을 탐색한다:

- 기존 `artifacts/spec.yaml`에서 관련 시나리오 읽기
- 기존 `artifacts/<feature>/spec.md` 확인
- 관련 컴포넌트의 현재 구현 확인

**판단 기준**: 사용자가 기존 기능명을 언급하거나, `artifacts/` 폴더가 존재하거나, `spec.yaml`에 매칭 ID 접두사가 있으면 기능 확장으로 판단한다. 완전히 새로운 기능이면 이 단계를 건너뛴다.

`$ARGUMENTS`에 대해 빈칸과 약한 부분을 찾아 **한 번에 하나씩** 질문합니다.

### 질문 규칙

- **한 번에 질문 하나.**
- `AskUserQuestion`을 사용하고, 가능하면 2-4개 구체적 선택지를 제시하세요.

### 질문 차원

필요한 순서대로 질문합니다. 모든 차원을 다 물어볼 필요는 없습니다.

1. **범위**
2. **엣지 케이스**
3. **성공 기준 구체화**
4. **전제 확인**
5. **도메인 고유 리스크**
   - 예: 결제 → "부분 환불 시 쿠폰 할인분은 어떻게 처리하나요?"

### 종료 조건

다음 중 하나면 질문을 멈추고 2단계로 이동합니다:

- 사용자가 중지 신호를 보냄
- 주요 빈칸이 모두 채워짐
- 3회 이상 질문한 후 새로운 발견이 없음

## 2단계: 문서 생성

템플릿은 `references/template.md`를 읽고 그 형식에 맞춰 작성하세요.

### 작성 규칙

- **구현 방법은 쓰지 마세요.**
- 모호한 부분은 1단계에서 모두 질문하세요. 사용자가 모른다고 답한 항목만 미결정 사항에 기록하세요.
- 성공 기준은 "이 입력이면 이 출력" 형식으로 작성하세요.
- 범위 제외 항목에는 **왜 빼는지** 이유를 함께 쓰세요.

### 문서 완성 체크리스트 (저장 전 검증)

문서를 저장하기 전에 다음을 확인하세요:

- [ ] 모든 성공 기준이 구체적 숫자/문자열로 입력 → 출력을 명시하는가
- [ ] 구현 기술(라이브러리, 언어, DB, 저장소)이 포함되지 않았는가
- [ ] 범위 제외 항목에 "왜 빼는지" 이유가 있는가
- [ ] 미결정 사항이 추측 대신 명시적으로 분리되었는가

## 3단계: 저장

최종 문서를 `artifacts/` 디렉토리에 저장합니다.

- 파일명: `artifacts/<feature-name>/spec.md`
- 저장 전 사용자에게 파일명 확인

## 4단계: spec.yaml 추출

spec.md 저장 후, 시나리오를 `artifacts/spec.yaml`에 구조화하여 추출합니다.

### 추출 규칙

1. `references/spec-schema.yaml`을 읽어 형식을 참조한다
2. 기존 `artifacts/spec.yaml`이 있으면 읽어 기존 시나리오를 파악한다
3. 기존 ID 목록을 확인하고 중복 시나리오를 식별한다
4. spec.md의 각 시나리오/성공 기준에서 구조화된 시나리오를 추출한다
5. ID는 `{FEATURE}-{NNN}` 형식 (피처명 대문자, 예: `KANBAN-001`). 기존 번호와 충돌하지 않도록 이어서 부여한다
6. `references/examples-guide.md`를 읽고 examples 작성 기준을 따른다
7. 기존 `artifacts/spec.yaml`이 있으면 scenarios에 append한다. 없으면 `version: 1`로 새로 생성한다
8. 사용자에게 추출 결과를 보여주고 확인 후 저장한다

### spec.yaml 검증 체크리스트 (저장 전 검증)

- [ ] input이 구체적 값인가 (변수나 플레이스홀더가 아닌 실제 값)
- [ ] expect가 화면에서 단언 가능한 값인가 (내부 상태/함수 호출 아님)
- [ ] given/when/then이 사용자 관점인가 (구현 용어 아님)
- [ ] ID가 기존 시나리오와 충돌하지 않는가
- [ ] 동일한 의미의 중복 시나리오가 없는가
- [ ] examples가 1개 이상인가
