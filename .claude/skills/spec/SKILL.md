# 요구사항 설계

요구사항 문서를 생성하고, 반복 질문으로 구체화합니다.

## 1단계: 초안 작성

`$ARGUMENTS`에 대한 요구사항 문서를 아래 형식에 맞춰 작성하세요.

```
## Overview
이 기능이 해결하는 문제를 1-2문장으로 설명

## 시나리오

### 1. 시나리오 제목
[상황] 사용자가 ~하면
[동작] 시스템이 ~한다

성공 기준:
- [ ] 구체적 입력 -> 기대 출력
- [ ] 구체적 입력 -> 기대 출력

### 2. 시나리오 제목
[상황] 사용자가 ~하면
[동작] 시스템이 ~한다

성공 기준:
- [ ] 구체적 입력 -> 기대 출력
- [ ] 구체적 입력 -> 기대 출력

(3-5개 시나리오)

## 전제 조건
- 전제 조건 나열

## 미결정 사항
- 결정이 필요한 항목
```

### 작성 규칙

- 구현 방법(어떤 라이브러리, 어떤 아키텍처)은 쓰지 마세요. "무엇을"에만 집중하세요.
- 모호한 부분은 추측하지 말고 미결정 사항에 나열하세요.
- 성공 기준은 "이 입력이면 이 출력" 형식으로 작성하세요.

## 2단계: 반복 질문

초안을 보여준 뒤, 빈칸과 약한 부분을 찾아 **한 번에 하나씩** 질문합니다.

### 질문 규칙

- **한 번에 질문 하나.** 절대 두 개 이상 질문하지 마세요.
- `AskUserQuestion`을 사용하고, 가능하면 2-4개 구체적 선택지를 제시하세요.
- 이전 답변을 기반으로 다음 질문을 만드세요.

### 질문 차원

필요한 순서대로 질문합니다. 모든 차원을 다 물어볼 필요는 없습니다.

1. **범위** -- 1차 출시에 포함할 것과 뺄 것
2. **엣지 케이스** -- 예외 상황에서 시스템이 어떻게 반응하는지
3. **성공 기준 구체화** -- 모호한 기준을 입력/출력으로 변환
4. **전제 확인** -- 암묵적으로 가정한 기술/환경 결정

### 종료 조건

다음 중 하나면 질문을 멈추고 3단계로 이동합니다:

- 사용자가 "됐어", "충분해", "저장해" 같은 신호를 보냄
- 주요 빈칸이 모두 채워짐
- 3회 이상 질문한 후 새로운 발견이 없음

## 3단계: 저장

질문 결과를 반영한 최종 문서를 `artifacts/` 디렉토리에 저장합니다.

- 파일명: `artifacts/<feature-name>/spec.md`
- 저장 전 사용자에게 파일명 확인
