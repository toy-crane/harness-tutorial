# 요구사항 설계

AI가 요구사항의 빈칸을 추측으로 채우면, 사용자 의도와의 간극이 구현 후에야 드러나고 되돌리는 비용이 크다.
반복 질문으로 변경 비용이 큰 결정을 구현 전에 사용자로부터 끌어내어, 그 간극을 최소화한다.

## 1단계: 반복 질문

### 사전 탐색 (기능 확장 시)

질문을 시작하기 전에, 기존 기능을 확장하는 경우 다음을 탐색한다:

- 기존 `artifacts/spec.yaml`에서 관련 시나리오 읽기
- 기존 `artifacts/<feature>/spec.md` 확인
- 관련 컴포넌트의 현재 구현 확인

**판단 기준**: 사용자가 기존 기능명을 언급하거나, `artifacts/` 폴더가 존재하거나, `spec.yaml`에 매칭 ID 접두사가 있으면 기능 확장으로 판단한다. 완전히 새로운 기능이면 이 단계를 건너뛴다.

### 빈칸 찾기: 사용자 흐름 따라가기

"범위는?", "보안은?", "성능은?" 같은 추상적 차원 순회는 구체적 결정으로 이어지지 않는다.
실제 사용자 흐름을 따라가야 변경 비용이 큰 빈칸이 자연스럽게 드러난다.

`$ARGUMENTS`에 대해, 사용자 흐름을 처음부터 끝까지 머릿속으로 시뮬레이션한다.
흐름을 따라가며 두 가지를 찾는다:

- **빈칸**: 정보가 없어서 다음 단계로 못 가는 곳 (여러 해석이 가능하거나 결정되지 않은 것)
- **갈림길**: 입력이 정상이 아닐 때 어떻게 되는지 정해지지 않은 곳 (빈 값, 중복, 초과 등)

빈칸과 갈림길 모두 변경 비용으로 분류한다:
- **변경 비용이 큰 것**: 반드시 질문한다.
- **변경 비용이 작은 것**: 기본값을 제안하고 넘어간다.

시뮬레이션 중 자연스럽게 드러나는 교차 기능 갭, 도메인 고유 리스크도 질문에 포함한다.

### 질문 규칙

- **한 번에 질문 하나.** (여러 질문을 한꺼번에 하면 사용자가 핵심 결정을 건너뛰는 경향이 있음)
- `AskUserQuestion`을 사용하고, 가능하면 2-4개 구체적 선택지를 제시하세요. (선택지가 없으면 질문이 모호해지고, 5개 이상이면 선택 피로가 생김)
- 이미 명시했거나 문맥에서 추론 가능한 것은 묻지 않는다.
- 기존 시나리오와의 교차점이 있으면 구체적으로 언급하며 질문한다.

### 종료 조건

다음 중 하나면 질문을 멈추고 2단계로 이동합니다:

- 사용자가 중지 신호를 보냄
- 사용자 흐름 시뮬레이션에서 더 이상 막히는 곳이 없음
- 3회 이상 질문한 후 새로운 발견이 없음

## 2단계: 문서 생성

템플릿은 `references/template.md`를 읽고 그 형식에 맞춰 작성하세요.

### 작성 규칙

- 모든 성공 기준이 구체적 숫자/문자열로 입력 → 출력을 명시하는가
- 구현 기술(라이브러리, 언어, DB, 저장소)이 포함되지 않았는가
- 범위 제외 항목에 "왜 빼는지" 이유가 있는가
- 미결정 사항이 추측 대신 명시적으로 분리되었는가 (사용자가 모른다고 답한 항목만 기록)

### 저장

- 파일명: `artifacts/<feature-name>/spec.md`
- 저장 전 사용자에게 파일명 확인

## 3단계: spec.yaml 추출

spec.md의 시나리오를 `references/spec-schema.yaml` 형식에 맞춰 `artifacts/spec.yaml`에 추출한다.

### 추출 규칙

- ID 형식: `{FEATURE}-{NNN}` (기존 번호와 충돌하지 않게 이어서 부여)
- examples 작성: `references/examples-guide.md` 기준을 따른다
- 기존 spec.yaml이 있으면 scenarios에 append한다. 없으면 `version: 1`로 새로 생성한다
- 사용자에게 추출 결과를 보여주고 확인 후 저장한다

### spec.yaml 검증 체크리스트 (저장 전 검증)

- [ ] input이 구체적 값인가 (변수나 플레이스홀더가 아닌 실제 값)
- [ ] expect가 화면에서 단언 가능한 값인가 (내부 상태/함수 호출 아님)
- [ ] given/when/then이 사용자 관점인가 (구현 용어 아님)
- [ ] 동일한 의미의 중복 시나리오가 없는가
- [ ] examples가 1개 이상인가

## 4단계: 독립 검토

spec.yaml 저장 후, 독립 리뷰어로 누락 시나리오를 검증한다.

### 실행

`spec-reviewer` 에이전트에 다음을 전달하여 독립 검토를 수행한다:
- 원본 요구사항: `artifacts/<feature>/spec.md`
- 추출된 시나리오: `artifacts/spec.yaml`

결과 처리:

**갭이 있는 경우:**
- 리뷰어가 제안한 시나리오 목록을 사용자에게 보여준다
- 사용자가 반영할 갭을 선택한다 (전체/일부/없음)
- 선택된 갭을 3단계의 추출 규칙·체크리스트를 적용하여 spec.md와 spec.yaml에 추가한다
- 기존 시나리오는 수정하지 않는다 (추가만 가능)

**갭이 없는 경우:**
- "독립 검토 완료: 누락 시나리오 없음"을 보고한다

## 5단계: 최종 확인

모든 단계 완료 후, 결과를 요약하여 사용자에게 보여준다.

- 생성된 파일 목록과 경로
- 총 시나리오 수
- 1단계에서 결정된 주요 사항 요약
